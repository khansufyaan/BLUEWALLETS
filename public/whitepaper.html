FIPS-Compliant Key Management for Regulated Financial Institutions

Blue Wallets Team —

Regulated financial institutions face a fundamental tension when managing digital assets. Existing solutions — MPC wallets, smart contract wallets, and cloud-hosted key management services — compromise on security by distributing or storing key material in cloud environments without FIPS or NIST certification. For banks operating under strict regulatory frameworks, this is unacceptable.

Blue Wallets is built for institutions that refuse to compromise. The platform is a white-label Wallet-as-a-Service (WaaS) solution deployed entirely on-premises, within the bank's own secure network. All cryptographic key material is generated and managed inside FIPS 140-3 Level 3 certified Hardware Security Modules — specifically, Thales Luna HSMs. The software, the portal, the API, and every microservice run on the institution's own infrastructure. Nothing leaves the building.

The platform implements BIP-32, BIP-39, and BIP-44 key ceremonies to derive hierarchical deterministic wallet structures inside the HSM. Child keys are wrapped (encrypted by the HSM) and stored externally in a database, enabling the platform to scale to millions of wallets without being constrained by HSM key slot limitations. When a transaction requires signing, the wrapped child key is retrieved from the database, unwrapped inside the HSM, used to sign, and the result is returned — the plaintext child key never exists outside the HSM boundary.

Design Principles

Blue Wallets Team —

The Problem with Existing Solutions

The digital asset custody market offers three dominant approaches, none of which satisfy the security requirements of regulated financial institutions.

MPC (Multi-Party Computation) wallets distribute key shares across multiple cloud servers. While no single server holds the complete key, the shares are recombined in memory during signing operations — creating a window where the full private key exists in volatile memory on cloud infrastructure. MPC implementations are not FIPS certified, and the cryptographic proofs underlying threshold signature schemes have not undergone the same decades of scrutiny as traditional ECDSA.1

Smart contract wallets implement access control at the blockchain protocol layer using multisig or account abstraction. While they eliminate the single-key-holder problem, they introduce smart contract risk — bugs in the contract logic can result in permanent loss of funds. Smart contract wallets are also chain-specific; there is no unified smart contract wallet standard across Bitcoin, Ethereum, and Solana.

Cloud-hosted KMS solutions (AWS CloudHSM, Azure Dedicated HSM) provide FIPS-certified hardware, but the institution does not control the physical infrastructure. For banks operating under regulations that mandate full custody of cryptographic material, a cloud provider's attestation is not equivalent to physical possession of the HSM in a bank-controlled data center.

Blue Wallets takes a different approach. The entire platform — software, keys, and infrastructure — runs on-premises, inside the institution's secure network, using HSMs that the institution physically owns and controls.

1. On-Premises, Not Cloud

Every component of the Blue Wallets platform is deployed within the financial institution's own infrastructure. The portal, the API gateway, the core microservices, and the database all run on servers inside the bank's network perimeter. The Luna HSM sits in the bank's data center, connected via a private network interface. No transaction data, no key material, and no customer information ever traverses the public internet.

This is not a hybrid model. There is no cloud fallback, no external dependency for signing operations, and no third-party service that can be subpoenaed for key material. The institution has complete sovereignty over the entire custody stack.

2. HSM-First Security

All cryptographic operations — master key generation, child key derivation, key wrapping, and transaction signing — happen exclusively inside the FIPS 140-3 Level 3 certified HSM. The application layer never sees plaintext private key material. It only handles wrapped (encrypted) keys, public keys, and signatures.

The HSM is a physically tamper-resistant device with its own processor, memory, and cryptographic accelerator. Keys created inside the HSM are marked with CKA_EXTRACTABLE = false at the PKCS#11 level. No API call, no administrative command, and no firmware update can extract the master key. Attempting to physically tamper with the device destroys all key material.2

3. BIP Standards for Scalability

The platform implements BIP-32 (hierarchical deterministic wallets), BIP-39 (mnemonic seed phrases), and BIP-44 (multi-account hierarchy) as key ceremonies executed inside the HSM. This solves a fundamental constraint: HSMs have a finite number of key slots (typically 2,000–100,000 depending on the model). A platform serving millions of wallets cannot store each wallet's private key as a separate HSM object.

Instead, the master key is generated and permanently stored in the HSM. Child keys are derived from the master key using BIP-32 derivation paths, wrapped (encrypted) by the HSM, and stored in an external database. The wrapped child keys are cryptographically useless without the HSM — they cannot be unwrapped, used for signing, or decrypted by any system outside the HSM boundary.

4. Policy Before Signing

Every transaction is evaluated against configurable policy rules before the HSM is invoked for signing. This is a hard architectural boundary: the signing service will not accept a request that has not passed through the policy engine. Spending limits, velocity controls, whitelists, time windows, and multi-level approval thresholds are enforced at the service layer, creating a programmable governance framework that each institution configures for their own compliance requirements.

5. Immutable Audit Trail

Every transaction — completed or rejected — records the HSM signature, the signed payload, all policy evaluation results, and any failure reasons. Rejected transactions are equally important for compliance: when a policy rule blocks a transfer, the rejection is recorded with the specific rule type and evaluation parameters. This enables compliance teams to demonstrate to regulators that controls are actively enforced, not merely configured.

Platform Architecture

Blue Wallets Team —

The entire platform runs on-premises within the financial institution's secure network. There are no external dependencies for any cryptographic operation. The architecture consists of four layers, each with distinct security boundaries. All inter-layer communication occurs over the bank's private network.

White-Label Portal and API

The institution receives a fully branded portal and API that they deploy for their own clients. The portal provides a web-based interface for vault management, wallet creation, transaction initiation, policy configuration, and role-based access control. The API provides the same functionality programmatically for institutions that integrate Blue Wallets into their existing systems.

Both the portal and API run on the institution's servers. The branding, domain, and authentication are fully controlled by the institution. End users interact with the bank's brand — Blue Wallets is invisible to the end customer.

Core Microservices

Four core services run as independent processes on the bank's server infrastructure, communicating through internal interfaces within the private network.

Key Service manages the BIP-32/39/44 key ceremony. It orchestrates master key generation inside the HSM, child key derivation, key wrapping, and the storage of wrapped keys in the database. It also handles key rotation and the extraction of public keys for address derivation. The Key Service is the only service that issues PKCS#11 commands to the HSM for key operations.

Wallet Service derives blockchain addresses from public keys across 9 supported chains. It manages vault-based wallet organization, balance tracking, and transfer execution. The Wallet Service never touches private key material — it works exclusively with public keys and wallet metadata.

Signing Service handles the transaction signing workflow. When a transfer is approved, the Signing Service retrieves the wrapped child key from the database, sends it to the HSM for unwrapping and signing via PKCS#11, and returns the signed transaction. The plaintext child key exists only inside the HSM for the duration of the signing operation.

Policy Engine evaluates all governance rules before any signing request reaches the HSM. Spending limits, velocity controls, whitelists, blacklists, time windows, and multi-level approval thresholds are enforced here. The Policy Engine has veto power — if any rule fails, the Signing Service is never invoked.

Without key wrapping, each wallet would consume one HSM key slot, capping the platform at the HSM's object limit. With BIP-32 derivation and key wrapping, only the master key and the wrapping key occupy permanent HSM slots. The wrapped child keys stored in the database are cryptographically inert. They are AES-256-wrapped by the HSM's internal wrapping key. Without access to the HSM, the wrapped blobs cannot be decrypted, used for signing, or converted into usable private keys. Even a complete database breach yields nothing actionable.

PKCS#11 Interface

All communication between the core services and the Luna HSM occurs through the PKCS#11 (Cryptoki) standard interface. PKCS#11 is the industry-standard API for hardware cryptographic tokens, defined by OASIS and used by every major HSM vendor. Blue Wallets uses the following PKCS#11 mechanisms:

HSM Cluster Topology

The Luna HSM is deployed in an active-active configuration with a hot standby node. Key material is replicated between active nodes using Luna's built-in cloning mechanism. The standby node receives asynchronous replicas and can be promoted in under 30 seconds during a failover event.

NodeRoleDescription
HSM Node 1ActivePrimary node for key ceremonies and signing

All three nodes are physically located in the institution's data center. Key IDs, labels, and wrapping keys are consistent across nodes, enabling transparent failover without re-deriving or re-wrapping child keys.

Security Model

Blue Wallets Team —

The security model is designed for a single assumption: any individual component can be compromised. Security does not depend on the integrity of any single server, service, database, or administrator. The architecture ensures that even a worst-case breach of the application layer yields no usable key material.

Threat Model

Compromised application server. An attacker with full control of the application server can access the database, which contains wrapped child keys. These wrapped keys are AES-256-encrypted by the HSM's internal wrapping key. Without the HSM, they cannot be decrypted. The attacker cannot issue signing requests because the policy engine runs as a separate service and requires valid approval chains.

Database breach. The database contains wrapped key blobs, public keys, wallet metadata, and transaction records. None of this data can be used to sign transactions or derive private keys. The wrapped key blobs are cryptographically inert without the HSM.

Malicious insider (Operator role). An operator can initiate transactions but cannot approve them. The Maker-Checker separation ensures that no single role can both create and authorize a transfer. The Compliance role is required to approve transactions before the HSM is invoked.

Physical HSM theft. Luna HSMs implement FIPS 140-3 Level 3 tamper-response mechanisms. Attempting to physically open or relocate the device without proper decommissioning triggers zeroization of all key material. The HSM becomes a brick.

Defense in Depth

Network perimeter. The entire platform runs inside the bank's private network. The HSM is accessible only from the core services subnet. There is no public internet exposure. mTLS is enforced for all service-to-service communication.

Application layer. JWT authentication on every request. Role-based access control with 28 granular permissions across 7 resource types (Vaults, Wallets, Keys, Policies, Roles, Users, AuditLogs). Maker-Checker enforcement — Operators initiate, Compliance approves, neither can do both. Input validation via Zod schemas on all API endpoints.

Cryptographic layer. Master key permanently resident in FIPS 140-3 Level 3 HSM. Child keys wrapped with AES-256 by the HSM's internal wrapping key. All signing operations occur inside the HSM boundary. Public key-only address derivation in the application layer. Transaction signatures provide cryptographic proof of authorization.

Data layer. Encryption at rest. No plaintext key material stored anywhere outside the HSM. Wrapped keys are useless without HSM access. Immutable audit logging for all state mutations. Separate read replicas for compliance and analytics queries.

Transaction Signing Lifecycle

Every transfer follows a deterministic pipeline. No step can be skipped.

Steps 1–3 are pre-signing gates. The API gateway validates the request schema. The Wallet Service verifies sufficient balance. The Policy Engine evaluates all attached policies. If any gate fails, the HSM is never invoked — the failure is recorded for audit.

Step 4 is the key unwrapping operation. The Signing Service retrieves the wrapped child key from the database and sends it to the HSM via C_UnwrapKey. The HSM decrypts the wrapped blob using its internal wrapping key, producing a temporary key object inside the HSM's secure memory.

Step 5 is the signing operation. The HSM signs the transaction hash using the unwrapped child key via C_Sign. The signing mechanism depends on the key algorithm — CKM_ECDSA for secp256k1 chains, CKM_EDDSA for Ed25519 chains. After signing, the temporary key object is destroyed inside the HSM.

Step 6 commits the result. The source wallet balance is decremented and the destination balance is incremented atomically. The transaction record is persisted with the signature, the full policy evaluation results, and a timestamp. The signed transaction is returned to the client.
